# 发版前真机验收表（≤10 项）

**续费 / 提醒 / 日期的完整设计链路**（改代码时也应主动对照）：[`RENEWAL-REMINDER-LIFECYCLE.md`](./RENEWAL-REMINDER-LIFECYCLE.md)

使用 **TestFlight 或 production profile** 安装的包（非 Expo Go）。iOS 为主；若带 IAP，用 **Sandbox** 测订阅。

---

## 1. 通知权限与提醒排程

- [ ] **首次**保存一条带「未来扣款日」的续费后，系统是否弹出通知授权；选「允许」后，能在 iOS **设置 › subradax › 通知** 中看到已开启。
- [ ] 故意在系统设置里**关掉**通知，回到 App 在设置里点「开启提醒」再走授权；保存/编辑一条续费后，确认没有误报「已排程」而又不提示需开权限（若当前实现静默不排程，心理上应能理解或与文案一致）。
- [ ] **改一条**续费的「下次扣款日」或「提前提醒天数」后，预期提醒时间应变（无需等到当天，可看「下次触发」逻辑是否重排：可依赖开发菜单或仅信代码 + 次日抽检）。
- [ ] **新建默认「提前 3 天」**：若把 **下次扣款日设成「今天」** 且不改成「当天提醒」，应理解 **名义提醒日已在三天前**，可能 **不再排今天/以后的通知**（与设计一致）；若产品期望例外，须见 `RENEWAL-REMINDER-LIFECYCLE.md` 中的特殊规则。
- [ ] **当天提醒 + 扣款日=今天**：保存后约 **1 小时**内有提醒（或之后打开 App 不无限推迟 — 锚点逻辑）；**错过当天 9:00** 但仍在同一日历日内应有 catch-up（见生命周期文档）。
- [ ] **扣款日已过、本地日期已进入次日**：冷启动或从后台回前台后，列表中的 **下次扣款日** 应变更为 **下一周期**（滚进）；通知应随 `reschedule` 更新。

## 2. 提醒文案与语言

- [ ] 系统在 **中文** 下，本地通知标题/正文为中文；切到 **English** 后新建或触发相关排程（或改语言后编辑列表触发 `reschedule`），通知用词为英文（与你们 i18n 设计一致即可）。

## 3. 列表维护（免费档）

- [ ] 条目数 **≤3** 时可正常新增、编辑、删除；删除后条数恢复可再加。
- [ ] 第 **4** 条应被拦截并引导 **Paywall**（`IAP_ENABLED=1` 时）；`IAP_ENABLED=0` 时则无限制（与当前构建一致即可）。

## 4. 订阅与 Pro（仅 iOS + IAP 开启时）

- [ ] Sandbox **购买**任一档（月/年）后，立即 **不限条数**、可添加第 4 条及以上。
- [ ] **恢复购买** 成功后恢复 Pro（删 App 重装后测一次更稳）。

## 5. 导出（Pro）

- [ ] 非 Pro 点导出 → 进 Paywall；Pro 点导出 → 系统分享 sheet 出现，文件为 **纯文本**，含名称、金额、周期、日期等（抽查内容完整）。
- [ ] 将导出文件存到「文件」或发到自己邮箱，确认 **可阅读、可留档**（换机安心的最小标准）。

## 6. 分享扩展 → 主应用（iOS）

- [ ] 在 Safari 或「邮件」中选一段文本 → **分享 › subradax** → 扩展界面 **`Open in subradax`** → 主应用打开 **来自邮件** 页且文本已进入粘贴区或已解析出候选（与当前产品一致即可）。

## 7. 粘贴 / 来自邮件（无 Gmail）

- [ ] **Gmail 关闭**（默认）时，页面 **无** Google 登录/扫信；粘贴一段含金额+订阅类关键字的文本可解析或给出「部分字段」提示（与文案一致）。
- [ ] （可选）`EXPO_PUBLIC_GMAIL_IMPORT_ENABLED=1` 的构建再测一遍 Gmail 全流程——首版若不上 Gmail 可跳过。

## 8. 边界与崩溃

- [ ] 快速连续：**加删改** 多条、切换 Tab、进出入 **Paywall**，无闪退。
- [ ] **飞行模式** 下仅操作本地列表与导出（不涉及商店恢复时可离线）；恢复购买需网络，失败时有合理提示。
